# warehouse-queue — вариант на Go

Альтернативная реализация бэкенда на Go (стандартная библиотека + драйверы БД). Переиспользует существующий фронтенд из `../public` (страница записи и кабинет менеджера).

## Статус и объём

Это **ядро паритета** — основной рабочий поток приложения, реализованный на Go:

- отдача статики (`index.html`, `manager.html`) из `../public`;
- склады, генерация и просмотр слотов (`/api/slots`), будни/выходные;
- бронирование (`POST /api/slots/:id/book`) с капчей, проверкой окна времени и защитой от двойной брони;
- капча, учёт посещений с разбором устройства/ОС/браузера (`/api/visit`);
- публичные настройки: запись без счёта, маскот, политика ПДн, политика cookie;
- кабинет: вход/выход, `/me`, список слотов, действия со слотами (взять/подтвердить/собрать/завершить/отменить/вернуть со сборки);
- настройки (журнал, выходные, маскот, тексты политик);
- управление менеджерами с признаком «администратор» (главный `admin` — админ всегда);
- две СУБД: **SQLite** (по умолчанию) и **PostgreSQL** — выбор через переменную `DB_BACKEND`.

Добавлено во втором заходе (`extras.go`):
- справочники с CRUD: классы машин и виды загрузки (вкл. публичные `/api/public/...` — форма записи работает целиком), кладовщики (с `has_pin`), категории, контрагенты, разрешённые подсети, забаненные телефоны/IP; в БД засеяны дефолтные классы/виды;
- бронирование пишет класс машины/вид загрузки и шлёт SMS (sms.ru) при наличии ключа в настройках;
- аналитика: `/api/manager/stats/timeseries` (минуты/часы/дни/недели/годы) и `/api/manager/stats/devices` (типы устройств, ОС, браузеры);
- резервные копии: скачивание/восстановление JSON, список/создание/восстановление автокопий, фоновый автобэкап с ротацией (интервал/кол-во — из настроек);
- проверка и применение обновлений через git (`/api/manager/check-update`, `/api/manager/update`);
- базовый кабинет кладовщика (`/api/storekeeper/slots`, действия по PIN).

Добавлено в третьем заходе (`extras2.go` + `main.go`):
- все эндпоинты вкладки «Настройки»: 1С (чтение/запись + переключатели allow-booking/warn-missing), SMS-ключ, Redis (конфиг + статус по TCP-пингу + тест), PostgreSQL (конфиг), TTL кэша, логотипы (заглушка), статус миграции;
- вкладки данных 1С и сопутствующие: сообщения, журнал проверок (check-logs), заказы 1С, заказы кабинета (c1-orders), номенклатура, менеджеры/инженеры 1С;
- отдача страницы кладовщика `/storekeeper` (`../private/storekeeper.html`);
- принудительный **IP-allowlist** для кабинета/кладовщика (loopback и пустой список разрешены).

Добавлено в четвёртом заходе (полный паритет):
- **активный Redis-кэш** (`redis.go`) — собственный мини RESP-клиент без зависимостей; публичные слоты кэшируются с TTL из настроек и инвалидируются при брони/отмене/миграции;
- **миграция и переключение SQLite↔PostgreSQL на лету** (`migrate.go`) — копирование всех таблиц в целевой бэкенд с пересинхронизацией SERIAL-последовательностей и переключением активного соединения; есть и «переключение без копирования»;
- **HTTP-валидация счёта в 1С при бронировании** (`onec.go`) — POST на настроенный URL с токеном, запись каждой попытки в `check_logs`, блокировка брони при недостоверном счёте (если админ это не разрешил);
- **агрегация водителей** — реальная сводка по `customer_phone` (имя, число записей, последняя дата);
- **загрузка логотипов тем** — GET/POST `logo_<тема>` в настройках (data URL).

> Примечание: переключение бэкенда на лету заменяет глобальное соединение; это административное действие на низкой нагрузке. На рестарте бэкенд снова берётся из `DB_BACKEND`. Контракт 1С реализован разумно по умолчанию (HTTP 200 + отсутствие `"valid":false`/`"found":false`/`"ready":false`) — при необходимости подгоните под ваш формат ответа 1С.

## Деплой

Отдельный скрипт `deploy/deploy-go.sh` (systemd, без Docker): ставит Go, клонирует/обновляет репозиторий, собирает бинарник, создаёт и запускает сервис `warehouse-go`. SQLite по умолчанию, либо `DB_BACKEND=postgres DB_DSN=... sudo bash deploy/deploy-go.sh`.

> ⚠️ В среде, где это писалось, **Go недоступен** (ни установить, ни скачать модули — всё блокируется), поэтому код **не проверен компилятором**. Соберите `go build ./...` у себя; если будут ошибки — пришлите вывод, поправлю.

## Запуск

```bash
cd go-service
go mod tidy          # подтянет драйверы (modernc.org/sqlite, jackc/pgx)
go build -o wq ./...

# SQLite (по умолчанию), фронтенд берётся из ../public:
./wq

# PostgreSQL:
DB_BACKEND=postgres DB_DSN='postgres://warehouse:warehouse@127.0.0.1:5432/warehouse?sslmode=disable' ./wq
```

Откройте `http://localhost:3000/` (запись) и `http://localhost:3000/manager.html` (кабинет, `admin`/`admin123`).

## Переменные окружения

| Переменная | По умолчанию | Назначение |
|-----------|--------------|-----------|
| `PORT` | `3000` | Порт сервера |
| `DB_BACKEND` | `sqlite` | `sqlite` или `postgres` |
| `DB_DSN` | `warehouse.db` (sqlite) | строка подключения |
| `STATIC_DIR` | `../public` | каталог со статикой фронтенда |

## Структура

```
go-service/
├── go.mod
├── main.go       # сервер, роутинг, сессии, генерация слотов, утилиты
├── db.go         # слой БД (SQLite/PostgreSQL), схема, настройки, сидинг
└── handlers.go   # HTTP-обработчики (публичные + кабинет)
```

Пароли хешируются SHA-256 (как в Node-версии), сессии — в памяти (cookie `wq_sess`). Дефолтный администратор `admin`/`admin123` создаётся автоматически — смените пароль.
