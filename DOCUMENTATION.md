# warehouse-queue — полная документация

Система онлайн-записи на склад с кабинетом менеджера, страницей кладовщика,
интеграцией с 1С, SMS-уведомлениями, кэшированием в Redis и поддержкой двух
СУБД (SQLite и PostgreSQL).

Проект существует в **трёх независимых реализациях с одинаковым функционалом и
общим фронтендом**:

| Вариант | Каталог | Технологии | Когда выбирать |
|---------|---------|-----------|----------------|
| **Node.js** | корень (`server.js`) | Node 20, Express, better-sqlite3 | Основной, самый зрелый; проще всего запустить |
| **Go** | `go-service/` | Go 1.21+, стандартная библиотека | Один статический бинарник, минимум зависимостей |
| **Rust** | `rust-service/` | Rust, tiny_http | Максимальная производительность одного бинарника |

Все три отдают один и тот же интерфейс из каталогов `public/` (страница записи,
кабинет) и `private/` (страница кладовщика), используют одинаковую схему БД и
одинаковые HTTP-эндпоинты. Выбирайте любой — поведение совпадает.

## Возможности

- **Онлайн-запись** на временные слоты по складам, с типами «мелкая» (шаг 15 мин)
  и «крупная» (шаг 30 мин) отгрузка; рабочее окно 10:00–17:30.
- **Защита от двойной брони** (атомарное условие `WHERE is_booked=0`), капча,
  ограничение «не раньше чем за 1 час» и «не дальше 2 недель».
- **Кабинет менеджера**: управление слотами (взять/подтвердить/собрать/
  завершить/отменить), справочники, менеджеры, аналитика, настройки.
- **Признак администратора** у менеджеров: вкладка «Настройки» доступна только
  админам. Главный `admin` — всегда администратор.
- **Страница кладовщика** с доступом по PIN-коду.
- **Интеграция с 1С**: проверка счёта/заказа и оплаты по HTTP, токен API,
  журнал проверок.
- **SMS-уведомления** через sms.ru.
- **Кэширование** свободных слотов в Redis с настраиваемым TTL.
- **Две СУБД**: SQLite (по умолчанию) и PostgreSQL, с миграцией/переключением.
- **Резервные копии**: ручные (скачать/восстановить) и автоматические с
  ротацией.
- **Аналитика**: посещения и брони по интервалам (минуты/часы/дни/недели/годы),
  разбивка по устройствам/ОС/браузерам.
- **Оформление**: светлая/тёмная/киберпанк/фэнтези + 4 сезонные темы, маскот
  (капля воды) с анимацией, баннер cookie, политики ПДн и cookie.
- **Обновление** из git прямо из кабинета (для native-установок).

## Быстрый старт

Самый короткий путь — Node на SQLite, локально:

```bash
git clone https://github.com/mnazarow/warehouse-queue.git
cd warehouse-queue
npm install
npm start
```

Откройте:

- запись — `http://localhost:3000/`
- кабинет — `http://localhost:3000/manager.html` (логин `admin`, пароль `admin123`)
- кладовщик — `http://localhost:3000/storekeeper`

> ⚠️ После первого входа **обязательно смените пароль `admin`** в кабинете
> (вкладка «Менеджеры»).

## Установка — все варианты

Во всех вариантах после установки доступны те же URL и тот же логин
`admin`/`admin123`.

### Node.js — вручную

Требования: Node.js 20+, npm. Опционально PostgreSQL и Redis.

```bash
git clone https://github.com/mnazarow/warehouse-queue.git
cd warehouse-queue
npm install
# по умолчанию SQLite в файле warehouse.db
PORT=3000 npm start
```

PostgreSQL и Redis в Node-варианте **подключаются и включаются из кабинета**
(вкладка «Настройки» → PostgreSQL / Redis) — приложение стартует на SQLite и
переключается на PostgreSQL по кнопке миграции. Предзаполнить параметры можно
через переменные окружения (`PGSQL_*`, `REDIS_*`) — см. раздел
[Переменные окружения](#переменные-окружения).

### Node.js — скрипт на чистый сервер

Ubuntu/Debian, от root. Скрипт ставит Node, готовит каталог, systemd-сервис.

```bash
# SQLite
sudo bash deploy/install-native.sh

# PostgreSQL по умолчанию (ставит и настраивает PostgreSQL)
sudo bash deploy/install-native-pg.sh
```

Подробности — в `INSTALL-NATIVE.md`.

### Docker

Полный стек (приложение + PostgreSQL + Redis + nginx + Let's Encrypt) одним
файлом `docker-compose.yml`:

```bash
cp .env.example .env
# отредактируйте .env: пароли БД, домен, e-mail для сертификата
docker compose up -d
```

Выпуск HTTPS-сертификата:

```bash
DOMAIN=zayavka.example.ru EMAIL=you@example.ru bash deploy/init-letsencrypt.sh
```

Полная инструкция и описание сервисов — в `DOCKER.md`.

### Go — на чистый сервер

Ubuntu/Debian, от root. Скрипт «под ключ»: ставит Go, PostgreSQL (создаёт БД и
роль), Redis, собирает бинарник, поднимает systemd-сервис `warehouse-go`.

```bash
# по умолчанию: PostgreSQL + Redis
sudo bash deploy/install-go-server.sh

# вариант на SQLite
sudo DB_BACKEND=sqlite bash deploy/install-go-server.sh

# свои параметры БД
sudo PG_DB=wh PG_USER=wh PG_PASSWORD='СильныйПароль' bash deploy/install-go-server.sh

# с nginx-прокси под домен
sudo WITH_NGINX=1 DOMAIN=zayavka.example.ru bash deploy/install-go-server.sh
```

Если Go и СУБД уже есть и нужно только собрать/обновить приложение —
`sudo bash deploy/deploy-go.sh`.

Сборка вручную:

```bash
cd go-service
go mod tidy && go build -o warehouse-go ./...
STATIC_DIR=../public PRIVATE_DIR=../private ./warehouse-go
```

### Rust — на чистый сервер

Ubuntu/Debian, от root. Скрипт «под ключ»: ставит Rust (rustup) и сборочные
зависимости (`build-essential`, `pkg-config` — нужны для встроенного SQLite),
PostgreSQL, Redis, собирает release-бинарник, поднимает сервис `warehouse-rs`.

```bash
# по умолчанию: PostgreSQL + Redis
sudo bash deploy/install-rust-server.sh

# вариант на SQLite
sudo DB_BACKEND=sqlite bash deploy/install-rust-server.sh

# с nginx-прокси под домен
sudo WITH_NGINX=1 DOMAIN=zayavka.example.ru bash deploy/install-rust-server.sh
```

Если Rust уже есть — `sudo bash deploy/deploy-rust.sh`.

Сборка вручную:

```bash
cd rust-service
cargo build --release
STATIC_DIR=../public PRIVATE_DIR=../private ./target/release/warehouse-rs
```

> Первая сборка Rust занимает несколько минут (компиляция зависимостей).

### Автодеплой через GitHub

В `.github/workflows/` настроены:

- `deploy.yml` — сборка Docker-образа и публикация в GHCR при push в `main`;
- `go-build.yml` — сборка и `go vet` Go-варианта;
- `rust-build.yml` — `cargo build --release` Rust-варианта.

На сервере автообновление организуется одним из способов:

- **git-pull** по таймеру/вебхуку — `deploy/auto-deploy.sh` (Docker) или
  `deploy/auto-deploy-native.sh` (native);
- **Watchtower** — следит за образом в GHCR и обновляет контейнер (профиль в
  `docker-compose.yml`).

Кнопки «Проверить обновления» и «Обновить» в кабинете (вкладка «Настройки»)
делают `git pull` и перезапуск для native-установок.

## Настройка

### Переменные окружения

**Node.js** (`server.js`):

| Переменная | По умолчанию | Назначение |
|-----------|--------------|-----------|
| `PORT` | `3000` | Порт сервера |
| `DB_PATH` | `./warehouse.db` | Путь к файлу SQLite |
| `SESSION_SECRET` | (генерится) | Секрет сессий — **задайте свой в проде** |
| `TRUST_PROXY` | — | `1`/`true`, если за обратным прокси (nginx) |
| `BACKUP_DIR` | рядом с БД | Каталог резервных копий |
| `REDIS_ENABLED` | `0` | Предвключить Redis (иначе из кабинета) |
| `REDIS_HOST`/`REDIS_PORT` | `127.0.0.1`/`6379` | Параметры Redis (сид) |
| `PGSQL_HOST/PORT/USER/PASSWORD/DATABASE` | — | Предзаполнение настроек PostgreSQL |
| `SMSRU_API_KEY` | — | Ключ sms.ru (можно задать в кабинете) |
| `SEED_CONNECTORS` | — | Демо-данные коннекторов 1С |

> В Node выбор СУБД делается **в кабинете** (миграция SQLite↔PostgreSQL). Env
> `PGSQL_*` только предзаполняют форму.

**Go** (`go-service`) и **Rust** (`rust-service`):

| Переменная | По умолчанию | Назначение |
|-----------|--------------|-----------|
| `PORT` | `3000` | Порт сервера |
| `DB_BACKEND` | `sqlite` | `sqlite` или `postgres` (выбор при старте) |
| `DB_DSN` | `warehouse.db` | Строка подключения (файл SQLite или `postgres://…`) |
| `STATIC_DIR` | `../public` | Каталог статики фронтенда |
| `PRIVATE_DIR` | `../private` | Каталог страницы кладовщика |
| `BACKUP_DIR` | `backups` | Каталог резервных копий |
| `SMSRU_API_KEY` | — | Ключ sms.ru (можно задать в кабинете) |
| `PG_HOST/PG_PORT/PG_DB/PG_USER/PG_PASSWORD` | — | Фолбэки для миграции в PostgreSQL |

> В Go/Rust бэкенд выбирается при старте через `DB_BACKEND`; миграция и
> переключение на лету тоже поддерживаются (из кабинета), но после рестарта
> берётся `DB_BACKEND`.

### Настройки в кабинете

Вкладка **«Настройки»** (только для администраторов):

- **О программе** — версия, время сервера, аптайм, нагрузка, число запросов к
  БД/Redis за час; кнопки «Проверить обновления» и «Обновить».
- **Аналитика** — графики посещений и броней, разбивка по устройствам.
- **Redis** — хост/порт/пароль/БД, включение кэша, кнопка «Тест».
- **PostgreSQL** — параметры подключения, миграция и переключение СУБД.
- **TTL кэша** — время жизни кэша по категориям (слоты, справочники и т.д.).
- **Резервное копирование** — скачать/восстановить, автобэкап (интервал и число
  хранимых копий), список копий.
- **Интеграция с 1С** — токен API, адрес сервера 1С, имя пользователя и
  **пароль** (поле видимое, с отдельной кнопкой «Сохранить»; в примерах запроса
  логин и пароль показываются открыто), адреса проверки заказа и оплаты,
  тумблеры «разрешить бронь без счёта / с недостоверным счётом / предупреждать».
- **Политики** — текст политики обработки ПДн и политики cookie.
- **Логотипы** — свои логотипы под темы.
- **Прочее** — журналирование действий, работа по выходным, включение маскота,
  разрешённые подсети (IP-allowlist для кабинета/кладовщика), баны телефонов/IP.

### HTTPS / домен

- **Docker**: nginx + certbot уже в `docker-compose.yml`; сертификат —
  `deploy/init-letsencrypt.sh`.
- **Native (Go/Rust/Node)**: запустите установщик с `WITH_NGINX=1 DOMAIN=…`,
  затем `apt install certbot python3-certbot-nginx && certbot --nginx -d ВАШ_ДОМЕН`.
- За прокси для Node задайте `TRUST_PROXY=1`, чтобы корректно определялись IP.

## Использование

### Страница записи

Адрес: `/`. Шаги для клиента:

1. Выбрать склад (если их несколько) и тип отгрузки (мелкая/крупная).
2. Выбрать класс машины и вид загрузки (справочники из кабинета).
3. Выбрать дату (будни; выходные — если включены в настройках) — отобразится
   сетка свободных слотов.
4. Выбрать слот, заполнить имя, телефон, при необходимости номер счёта,
   организацию, комментарий.
5. Поставить галку согласия на обработку ПДн, решить капчу, подтвердить.

После брони клиенту уходит SMS (если задан ключ sms.ru). Свободные слоты
кэшируются в Redis (если включён) и обновляются при бронях и отменах.

### Кабинет менеджера

Адрес: `/manager.html`. Вход — логин/пароль менеджера.

- **Слоты / Заявки** — фильтр по дате/типу/складу; по каждой брони действия:
  взять в работу, подтвердить, отправить на сборку, завершить, отменить,
  вернуть со сборки. Виден клиент, телефон, счёт, организация, комментарий.
- **Кладовщики** — список с PIN-кодами (PIN не отображается, только признак
  «задан»).
- **Справочники** — классы машин, виды загрузки, категории, контрагенты.
- **Менеджеры** — создание/редактирование, признак «администратор» (главного
  `admin` нельзя разжаловать или удалить).
- **Аналитика** — посещения и брони по интервалам, устройства/ОС/браузеры,
  водители.
- **1С / данные** — заказы 1С, журнал проверок, сообщения, номенклатура.
- **Настройки** — см. раздел [Настройки в кабинете](#настройки-в-кабинете).

### Страница кладовщика

Адрес: `/storekeeper`. Кладовщик вводит **PIN-код** (заданный менеджером) и
видит активные и завершённые отгрузки; может отметить «на сборку» и «завершено».
Доступ к странице дополнительно ограничивается списком разрешённых подсетей
(если он задан в настройках).

### Интеграция с 1С

1. В кабинете → «Настройки» → «Интеграция с 1С»: сгенерируйте **токен API**,
   укажите адрес сервера 1С, имя пользователя и **пароль** (видимое поле,
   кнопка «Сохранить»).
2. Укажите **адрес проверки заказа** и **адрес проверки оплаты/готовности**.
   Под каждым полем показывается пример запроса (метод, URL, заголовок
   `Authorization: Basic логин:пароль`, тело с `invoice_number`).
3. Кнопкой «Тестировать запрос» проверьте связь, указав номера счетов.
4. Тумблеры определяют, можно ли бронировать без счёта или с недостоверным
   счётом и нужно ли предупреждать.

При бронировании, если задан адрес проверки заказа, бэкенд обращается к 1С,
пишет результат в журнал проверок и блокирует бронь при недостоверном счёте
(если это не разрешено тумблером).

## Обслуживание

**Управление сервисом (native, systemd):**

```bash
systemctl status warehouse-go     # или warehouse-rs / warehouse (Node)
systemctl restart warehouse-go
journalctl -u warehouse-go -f      # логи в реальном времени
```

**Docker:**

```bash
docker compose ps
docker compose logs -f app
docker compose pull && docker compose up -d   # обновление образа
```

**Резервные копии.** Скачивайте дамп из кабинета или настройте автобэкап
(интервал + число хранимых копий). Файлы — в `BACKUP_DIR`. Восстановление — из
кабинета (загрузка дампа) или выбором автокопии.

> Дамп содержит **секреты** (токен 1С, пароли PostgreSQL/Redis) — храните копии
> в защищённом месте.

**Миграция SQLite↔PostgreSQL.** В кабинете → «Настройки» → PostgreSQL: задайте
параметры и нажмите «Мигрировать» (копирует данные и переключает) или
«Переключить» (без копирования). Для Go/Rust после рестарта используется
`DB_BACKEND`, поэтому зафиксируйте выбор в переменных окружения сервиса. После
ручной миграции в PostgreSQL при необходимости выполните
`deploy/fix-pg-sequences.sql` для пересинхронизации счётчиков id.

**Обновление.** Native: кнопка «Обновить» в кабинете (git pull + рестарт) либо
`deploy/auto-deploy*.sh`. Docker: pull нового образа или Watchtower.

## Безопасность

- Смените пароль `admin` сразу после установки.
- Задайте `SESSION_SECRET` (Node) и сильные пароли БД.
- Ограничьте доступ к кабинету и кладовщику по IP (настройка «Разрешённые
  подсети»). На loopback и при пустом списке доступ открыт.
- PIN-коды кладовщиков и пароли менеджеров не отдаются клиенту.
- Резервные копии содержат секреты — защищайте их.
- Включайте HTTPS (см. [HTTPS / домен](#https--домен)); за прокси — `TRUST_PROXY=1`.
- Капча и лимит частоты броней снижают автоматизированный спам.

## Устранение неполадок

- **Слоты не отображаются** — проверьте, что выбран будний день (или включена
  работа по выходным), и что склад создан. Слоты создаются «лениво» при запросе.
- **PostgreSQL: `column … does not exist` / `duplicate key`** — после миграции
  выполните `deploy/fix-pg-sequences.sql`; убедитесь, что схема создана.
- **Redis не кэширует** — включите Redis в «Настройках» и нажмите «Тест»; кэш
  работает только при `redis_enabled=1` и доступном сервере.
- **SMS не приходят** — задайте ключ sms.ru (кабинет или `SMSRU_API_KEY`).
- **Доступ к кабинету «Access denied from this IP»** — ваш IP не в списке
  разрешённых подсетей; добавьте подсеть или очистите список.
- **Go/Rust не собираются** — выполните `go build ./...` / `cargo build` и
  пришлите вывод; для Rust нужен C-компилятор (`build-essential`).
- **Порт занят** — измените `PORT`.

## Ограничения и заметки

- Go и Rust выбирают СУБД при старте через `DB_BACKEND`; переключение на лету
  доступно, но не сохраняется между перезапусками.
- Контракт проверки 1С: `POST` с HTTP Basic-аутентификацией (логин/пароль 1С) и
  телом `{ "invoce_number": ["<счёт>"] }` (поле названо именно так, с опечаткой —
  как в API 1С). Ответ разбирается по объекту `results`: счёт считается валидным,
  если его `status` начинается с `found`/`найден`. Если поля `results` нет или
  ответ не разобрать — проверка пропускается (бронь не блокируется). Одинаково
  во всех трёх вариантах.
- Сервер Rust однопоточный (один цикл обработки) — этого достаточно для
  внутреннего складского приложения; внешние вызовы (1С/SMS) имеют таймауты.
- Дефолтные учётные данные `admin`/`admin123` обязательно меняйте.
