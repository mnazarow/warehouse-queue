# warehouse-queue — вариант на Rust

Альтернативная реализация бэкенда на Rust. HTTP — на `tiny_http` (блокирующий,
один цикл обработки запросов), хранилище — SQLite (`rusqlite`, bundled) или
PostgreSQL (`postgres`) за единым перечислением `Db`. Переиспользует
существующий фронтенд из `../public` и страницу кладовщика из `../private`.

## Объём (паритет с Node/Go-версиями)

- статика (`index.html`, `manager.html`) + страница кладовщика `/storekeeper`;
- склады, генерация и просмотр слотов, будни/выходные;
- бронирование с капчей, окном времени, защитой от двойной брони, проверкой
  счёта в **1С** (если настроен URL) и отправкой **SMS** (sms.ru);
- учёт посещений с разбором устройства/ОС/браузера;
- кабинет: вход/выход, профиль, слоты и действия со слотами;
- настройки: журнал, выходные, маскот, тексты политик ПДн/cookie;
- менеджеры с признаком «администратор» (главный `admin` — админ всегда);
- справочники с CRUD: классы машин, виды загрузки, категории, контрагенты,
  кладовщики, разрешённые подсети, забаненные телефоны/IP;
- аналитика: временные ряды (минуты/часы/дни/недели/годы) и устройства;
- резервные копии: скачивание/восстановление, список/создание автокопий,
  фоновый автобэкап с ротацией;
- проверка/применение обновлений через git;
- **активный Redis-кэш** (свой мини RESP-клиент без зависимостей) для публичных
  слотов с TTL из настроек и инвалидацией при брони/отмене/миграции;
- **миграция и переключение SQLite↔PostgreSQL на лету** с пересинхронизацией
  SERIAL-последовательностей;
- настройки вкладки «Настройки»: 1С, SMS, Redis (конфиг+статус+тест), PostgreSQL,
  TTL кэша, логотипы; вкладки данных (журнал проверок, сообщения, номенклатура,
  заказы кабинета, водители);
- принудительный **IP-allowlist** для кабинета/кладовщика.

## Особенности реализации

Параметры запросов инлайнятся в SQL как экранированные литералы (через единую
функцию `lit`), что даёт одинаковый путь для обоих драйверов и снимает проблему
сопоставления типов. Любой пользовательский ввод проходит через экранирование.

Сервер однопоточный (один цикл `incoming_requests`), что исключает гонки и
упрощает переключение БД на лету. Для внутреннего складского приложения этого
достаточно; долгие внешние вызовы (1С/SMS) имеют таймауты.

> ⚠️ В среде, где это писалось, **Rust/cargo недоступен** (как и сеть для
> скачивания crate'ов), поэтому код **не проверен компилятором**. Соберите
> `cargo build` у себя; если будут ошибки сборки — пришлите вывод, поправлю.

## Сборка и запуск

```bash
cd rust-service
cargo build --release       # подтянет tiny_http, rusqlite(bundled), postgres, ureq, chrono...

# SQLite (по умолчанию), фронтенд из ../public:
STATIC_DIR=../public PRIVATE_DIR=../private ./target/release/warehouse-rs

# PostgreSQL:
DB_BACKEND=postgres \
DB_DSN='postgres://warehouse:warehouse@127.0.0.1:5432/warehouse?sslmode=disable' \
STATIC_DIR=../public PRIVATE_DIR=../private ./target/release/warehouse-rs
```

Откройте `http://localhost:3000/` (запись) и `http://localhost:3000/manager.html`
(кабинет, `admin`/`admin123` — смените пароль).

## Переменные окружения

| Переменная   | По умолчанию | Назначение |
|--------------|--------------|-----------|
| `PORT`       | `3000`       | Порт сервера |
| `DB_BACKEND` | `sqlite`     | `sqlite` или `postgres` |
| `DB_DSN`     | `warehouse.db` (sqlite) | строка подключения |
| `STATIC_DIR` | `../public`  | каталог со статикой |
| `PRIVATE_DIR`| `../private` | каталог со страницей кладовщика |
| `BACKUP_DIR` | `backups`    | каталог резервных копий |
| `SMSRU_API_KEY` | —         | ключ sms.ru (можно задать и в кабинете) |

## Деплой

Отдельный скрипт `deploy/deploy-rust.sh` (systemd, без Docker): ставит Rust
(rustup), сборочные зависимости (`build-essential` для bundled SQLite),
клонирует/обновляет репозиторий, собирает release-бинарник и поднимает сервис
`warehouse-rs`. SQLite по умолчанию, либо
`DB_BACKEND=postgres DB_DSN=... sudo bash deploy/deploy-rust.sh`.
